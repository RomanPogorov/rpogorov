                        System · 2024 Q3 — ongoing · 4 layers, multi-product
                        System
                    
        
                    
        Design system from Figma to Storybook
        
                    
        Multi-layer design system on top of shadCN with semantic tokens, custom states and bidirectional Figma ↔ code synchronization. Used in production by all product teams.
        
                    
                    
        
                        
        shadCN was chosen as the base — it provides components and freedom without imposing a visual language. But shadCN is "free": one team takes a button and adds Tailwind utilities to make their version, another team makes their own. With multiple products and teams, this freedom needed limits.
        
                        
        The solution — a semantic layer underneath shadCN:
        
                        
          
                            
        • Unified typography styles
          • 
                            
        • Semantic color layer
          • 
                            
        • Rewritten components (with Claude) — our state set, additional states not present in shadCN out of the box, enriched inputs and selects
          • 
                            
        • Storybook — also serves as documentation
          • 
                        
        
                        
        Architecturally: teams use only our components with our tokens. shadCN remains the foundation, but not the extension point — extension happens only through the semantic layer.
        
                        
        Bidirectional Figma ↔ Code pipeline
        
                        
        Components were built in code first, then returned to Figma. Opposite of the usual order, and intentional: code is the source of truth, Figma is the representation.
        
                    
        
                    
        
                        
                        
                    
        
                    
                    
        
                        
        
                            
        All.
        
                            
        Product teams use it in production
        
                        
        
                        
        
                            
        Public
        
                            
        Figma library (link TBD)
        
                        
        
                        
        
                            
        shadCN

                        Tools
                    
        
                    
        Figma ↔ Code mapping skill
        
                    
        Custom Claude skill that takes a Figma interface as input, maps it to design-system components, identifies states, builds a correspondence table, and asks clarifying questions when uncertain. Used by developers in production.
        
                    
                    
        
                        
        Built when no similar tools existed. The pipeline:
        
                        
          
                            
        • Designer hands the assembled Figma interface to Claude.
          • 
                            
        • Claude scans the registry, sees what components are available, what maps to what it sees, what states each component should use.
          • 
                            
        • Builds a correspondence table: what was found, what was mapped, which state where.
          • 
                            
        • If anything is unclear — asks a question.
          • 
                        
        
                        
        First-run accuracy was 70–80%. Sometimes wrong on state, padding, or layout. Iteratively improved instructions — quality grew. Now developers use the skill in their work.

        Landing platform & AI pipeline
        
                    
        Turned landing-page production from manual and chaotic (3 months per landing through C-level alignment) into infrastructure: any non-tech employee drops Claude any text format and gets a brand-styled landing in 2–3 days.
        
                    
                    
        
                        
        The CTO said: "Guys, this is bullshit. Let's automate this." So I started building the system step by step.
        
                        
        v1 — generator on a block library
        
                        
          
                            
        • Block library in code, designed and assembled by me
          • 
                            
        • Claude skill trained on the library
          • 
                            
        • Accepts MD / Word / HTML / PDF / archives — any input
          • 
                            
        • Locally generates a landing in 10–20 minutes
          • 
                            
        • ACP feature "Fixik" — DOM inspector + Claude chat panel through Chrome CLI for live tweaks
          •

                        Process shift · ~1 week → ~1 hour · 30+ presentations generated
                        Process shift
                    
        
                    
        Presentation generator
        
                    
        Same approach as the landing platform, applied to presentations. Two entry points (drop-and-get + visual editor), three generation modes (Normal / Variety / Madness). About 30 presentations generated, team in love.
        
                    
                    
        
                        
        The problem: lots of sales pitches, lots of presentations. Everyone made them in different tools (Google Slides, Figma, PowerPoint, online services), with different styles. No system, no shared style constants, no reuse.
        
                        
        Took about a month to build.
        
                        
        Entry point 1 — drop-and-get
        
                        
        Drop MD / text / HTML / PDF into Claude → "make a presentation using Presentation Builder Skill". Three modes:
        
                        
          
                            
        • Normal — built on the block registry. The skill looks at registered blocks and tries to use them. Everything anchored to design tokens, design system, single style.
          • 
                            
        • Variety — for non-standard input. Generates from content, not from blocks. Still in our style, but inventive — leverages base constants without imitating predefined blocks.
          • 
                            
        • Madness — total goof-around. Whatever it wants. Mostly for fun, but it's there.
          • 
                        
        
                        
        Entry point 2 — visual editor
        
                        
        Slides + modules + JSON structure → Claude renders a controlled solution. For people who like control: ready page patterns, different elements (text, icons, text-with-icon, cards, dividers), per-slide data import.
        
                        
        The structure is flexible: each module is a JSON node with fields, the visual representation lets you edit content right there or download generated JSON, edit it in your own editor, and feed it back to Claude.

                        Tools · PostHog inside every page · zero context switch
                        Tools
                    
        
                    
        Embedded analytics on every page
        
                    
        A "Stats" button visible only to logged-in employees on every page of the ecosystem — landing, article, case study, documentation. Click opens a drawer with PostHog data sliced for that specific page. No tool switching, no searching for the page in a global list.
        
                    
                    
        
                        
        Standard analytics workflow: open separate tool → find the page in a list of all pages → load the dashboard for it. For a company with hundreds of articles, landings, case studies, and docs — that's friction.
        
                        
        The fix: every page knows its own analytics scope. Logged-in employees see a small "Stats" button. Click — drawer slides in with PostHog data filtered to that page: traffic, sources, behavior, conversions, multiple slices.
        
                        
        The cultural impact — analytics became part of daily browsing. People look at performance data while reading their own articles or landings. Authors see their readership without effort.

        • Pull the team's repo.
          •

          •

        JSON Parser & vibe-code as method
        
                    
        From day one, complex tasks (more than a single screen or simple flow) were assembled in code on React. Working demos instead of static mockups. JSON Parser was one of the visible artifacts — a UI hypothesis prototype for Aidbox, used to validate direction and present to the team.
        
                    
                    
        
                        
        Vibe-code became the default way to validate hypotheses. Static mockups skip questions that show up only in interaction — performance, edge cases, real keyboard input, scroll behavior. Working prototypes catch them.
