      CS/01 Americor Lead Product Designer 2023 — 2025

      Redesigning how Americans escape debt — a multi-brand fintech rebuild.

      Americor is a fintech in the debt-relief space — a web app plus iOS and Android. I was brought in as Lead PD to bring order to the design process and run a full product redesign ahead of IPO.

      // TL;DR
      • Designed and shipped a multi-layer design system across three platforms (iOS / Android / Web): root → semantic → brand → platform. Covered ~50–60% of interfaces and gave Americor's multiple brands one shared visual language.
      • Flipped the process from mockup → handoff → dev to code-first: in the final ~3 months the team worked on a live React prototype + Storybook; new features got assembled directly from code.
      • To back redesign decisions with real signal, I built a user-interview pipeline and a custom analyzer for the Facebook community feed — decisions ran on data, not assumptions.
      01 — Start

      What I walked into, and where I started

      The team was middle- and junior-level designers. There was no actual design system. Reusable elements lived scattered across files — never formalized as components. Every new mockup got hand-assembled from disparate pieces. Maintaining or updating interfaces at scale was impossible this way.

      An earlier attempt to bring in a freelancer to define the brand had stalled — six months in, the work was 40–50% done and the designer left. Engineers were spending weeks hunting for a "source of truth" for components. Visual consistency across platforms didn't exist.

      I started with an audit: pulled every screen across desktop + responsive, iOS and Android, surfaced the recurring patterns, and decided what had to get systematized first.

      image · audit landscape (before)
      Audit of the existing mockups — which patterns to unify, which to leave alone.
      02 — Foundation

      A multi-layer design system

      Built on the audit, I designed and shipped a multi-layer token architecture:

      • Root — primitive values: colors, spacing, typography.
      • Semantic — a layer of meaningfully-named tokens.
      • Brand — a brand layer: Americor has subsidiaries in different states, each with its own color identity driven by per-state regulation.
      • Platform modes — Android, iOS, Web — managed from one place.

      In parallel I reorganized the whole Figma library: iterations, branches for development, clear navigation across mockups. Together with the teams we started rolling the system out component by component and reached ~50–60% interface coverage.

      image · token architecture diagram (root → semantic → brand → platform)
      The four token layers and three platform modes, on one diagram.
      image · figma library structure
      Figma library reorg: branches, iterations, navigation.
      03 — Pre-IPO

      The pre-IPO redesign

      Mid-rollout, leadership decided on a full redesign of the mobile and web apps — the company was preparing for IPO and the existing design had aged out.

      By that point the middle designer had left and we were down to two — me leading and a junior. I drove the redesign end-to-end and was the quality bar for everything that shipped.

      To ground decisions in data rather than assumptions, I built a feedback pipeline: ran user interviews and, in parallel, launched a custom analyzer for the Facebook community feed (see next section).

      image · before / after key screens
      Before / after of key screens from the redesign.
      04 — Tools

      Vibecoding: custom tools to back decisions

      Facebook community analyzer. I wrote a scraper that pulled every post and comment from Americor's Facebook community, then ran each message through an LLM that classified topic (UX, support, a specific feature) and sentiment (complaint, gratitude, request, frustration). On top of that I built a UI with filters by topic and sentiment — the team could surface problem areas in seconds and ground redesign decisions in real signal.

      NSF Shortfall simulator. While designing a tricky shortfall-prediction feature for payment dates, I ran into gnarly business logic. I built an interactive simulator end-to-end myself — it let me reason through the mechanics, verify edge-case scenarios, and ship the design faster.

      image · feedback analyzer UI (filters by topic + sentiment)
      Feedback analyzer — filters by topic and sentiment.
      image · NSF shortfall simulator
      Simulator for shortfall business logic on payment dates.
      05 — Shift

      Code-first: live prototypes instead of mockups

      In the final stretch (~3 months) I moved the entire design process to code-first: rebuilt every app screen — including the responsive set — as a real React app. Alongside it lived a Storybook where every component shipped from. The interface was assembled from those components, end of story.

      I built new features almost entirely in code with Claude, then generated Figma components from the finished code for engineering handoff. Hypotheses were captured and tested on a real, working prototype — not static mockups.

      It flipped the process: from "design → handoff → dev" to "prototype in code → validate → hand over ready components."

      CS/02 Health Samurai Senior Product Designer 2024 — present

      A design system for five healthcare products, half of it generated by AI.

      Health Samurai is a healthcare-IT company building FHIR-native infrastructure: Aidbox, Aidbox Forms, Audit Box, MPI Box, MDM Box. Flat org — no QAs, no PMs, only senior T-shape engineers and Friday seminars where designers and engineers cross-train.

      // TL;DR
      • Stood up a shadCN-based design system end-to-end — Figma + code + Storybook in lockstep, plus a Claude skill that pulled components straight out of Figma before that tooling existed elsewhere.
      • Built a markdown-first landing platform with an in-browser MCP chat: product owners draft in MD, point-and-edit in the live DOM, and ship in days. Time-to-landing dropped from 1–1.5 months to 3–4 days.
      • Generators on top of the same primitives — presentation generator for any teammate, kudos image generator trained on the brand mascot, all wired through the design system.
      • Vibecoded the FireCamp conference app in a week — voting, schedule, subscriptions. It became the proof point that convinced C-level the code-first approach was real.
      01 — Start

      Joining as the second designer

      I joined Health Samurai as the second designer. The brief was wide: stand up the design function, ship a system, redesign the existing product line, and lay AI-first design infrastructure. The company was already pointed that way — I started experimenting with code-first from day one.

      The starting state was honest: every product had been built by engineers without a designer. They worked, but carried real UX/UI debt. There was no design system. Landings were authored by C-level — content duplicated, scattered across pages, and unfocused. The blog and docs were just as scattered.

      A single landing took 1–1.5 months end-to-end, mostly because the approval chain was long and the surface to edit was hand-rolled.

      // the team

      Flat org by design: no QAs, no PMs, only senior T-shape engineers. Every Friday the company runs internal seminars — designers teach engineers visual fundamentals, engineers teach designers how to read code. Cross-training is the culture, not a perk.

      02 — System

      A design system on shadCN, mirrored in Figma and code

      Built the design system on shadCN — not raw Tailwind, but a styled, extended layer with custom components for tighter control. Each component lived twice: in Figma and in code. Every one of them shipped through Storybook for engineering review before landing in production.

      The Claude skill pulled components straight out of Figma — at a moment when this kind of tooling didn't exist yet.

      I iteratively tightened the prompt so the agent parsed component anatomy correctly, and set up a sync script: a change on either side propagated to the other in seconds. One source of truth, two surfaces.

      04 — Platform

      Landings: from 1.5 months to 3–4 days

      The old way was a chain: brief, design, review, dev, review, ship — each handoff burning days. I rebuilt the surface around markdown and a small set of agents.

      • Step 1. Product owner writes content in Markdown.
      • Step 2. System auto-assembles a draft landing from the MD.
      • Step 3. Owner uses an in-browser MCP chat to point at any DOM element and say what to change — live.
      • Step 4. Designer polishes; landing ships in 3–4 days.

      All legacy landings ran through this pipeline — single visual style, one place to edit.

      1.5mo → 3–4d
      Time to ship a landing — from a 1–1.5-month chain to 3–4 days end-to-end.

      05 — Generators

      Decks, blocks, and a kudos image generator

      A block library for non-tech. Built a library of blocks and "organisms" any non-technical employee can use. They open the system, snap a page together like Lego, fill the copy, and download an MD or hand it to an agent.

      Presentation generator. On top of the same primitives I built a deck generator. Anyone — sales, support, tech — drops in an MD file and gets a fully styled presentation in the company's identity. It quietly killed the chronic shortage of decent decks across the org.

      Kudos image generator. Health Samurai runs an internal "kudos" currency through a Telegram bot. Wrote a generator that takes the gratitude text and an LLM trained on the brand's mascot — and renders the mascot acting out the scene. Every kudos gets its own picture.
